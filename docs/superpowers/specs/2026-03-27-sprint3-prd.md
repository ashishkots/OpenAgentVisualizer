## 1. Executive Summary

Sprint 1 delivered a complete backend platform (auth, agent CRUD, event ingestion, OTLP traces, gamification, metrics, alerts, WebSocket). Sprint 2 transformed that backend into a visual product: PixiJS 2D canvas with 50+ agent sprites at 60fps, XState finite state machines driving avatar animations, ReactFlow topology graphs, GSAP animated transitions, seven dashboard pages, WebSocket room subscriptions via Redis Pub/Sub, session replay, an achievement system with 10 achievements, and 166+ test cases across frontend and backend.

Sprint 3 extends OpenAgentVisualizer along three axes:

1. **Immersive 3D Viewer (OAV-301)** -- Integrate an Unreal Engine 5 Pixel Streaming viewer into the web application. Agents appear as 3D avatars in a navigable virtual environment. The UE5 application communicates bidirectionally with the FastAPI backend via WebSocket for real-time state synchronization. When a GPU server is unavailable, the existing 2D PixiJS canvas serves as an automatic fallback.

2. **Cross-Product Integrations (OAV-302 through OAV-305)** -- Connect OpenAgentVisualizer to four Open* Suite products. OpenTrace provides distributed trace data for waterfall visualization. OpenMesh provides multi-agent workflow topology for mesh graph rendering. OpenMind provides knowledge graph data for relationship exploration. OpenShield provides security posture data for compliance dashboards. Each integration follows a consistent pattern: backend adapter service, configuration management, frontend dashboard panel, and graceful degradation when the partner product is unreachable.

3. **CLI Plugin and DevOps (OAV-306, OAV-235, OAV-241-243)** -- A terminal-based plugin for Claude Code and Codex that surfaces agent status, streams events, triggers actions, and renders ASCII metric charts. Deferred DevOps tasks from Sprint 2 (GitHub Actions CI, production Dockerfiles, Prometheus metrics endpoint, Playwright E2E tests) are completed.

### OAV-301: Unreal Engine 5 3D Viewer Integration

**User Story:** As an AI team lead, I want to view my AI agents as 3D avatars in an immersive virtual environment rendered by Unreal Engine 5, so that I can spatially navigate and inspect agent activity in a way that feels more intuitive and engaging than a 2D canvas.

**Description:** The UE5 3D viewer uses Pixel Streaming to deliver a real-time 3D scene to the browser. A signaling server coordinates WebRTC connections between the UE5 application and the browser. The React frontend embeds the stream in an iframe or WebRTC video element. Bidirectional communication flows through a dedicated WebSocket channel: the web app sends navigation commands and selection events to UE5, and UE5 sends camera state and object interaction events back to the web app. When no GPU server is available (or the Pixel Streaming connection fails), the system automatically falls back to the existing 2D PixiJS canvas within 3 seconds.

**Acceptance Criteria:**

- **AC-1:** Given a running UE5 Pixel Streaming server, when a user navigates to the 3D view tab, then a WebRTC video stream renders in the browser showing the 3D virtual environment with agent avatars.
- **AC-2:** Given agents exist in the workspace, when the 3D scene loads, then each agent appears as a distinct 3D avatar positioned in the environment, with a name label, level indicator, and status color matching the agent's current state.
- **AC-3:** Given the 3D viewer is active, when the user clicks on an agent avatar in the 3D scene, then the web app receives a selection event and displays the agent detail sidebar (same data as the 2D tooltip).
- **AC-4:** Given the 3D viewer is active, when an agent's state changes via WebSocket event, then the UE5 scene updates the avatar's animation and visual indicators within 500ms.
- **AC-5:** Given the UE5 Pixel Streaming server is not running, when a user navigates to the 3D view tab, then the system detects unavailability within 3 seconds and automatically switches to the 2D PixiJS canvas with an informational banner: "3D viewer unavailable -- showing 2D view."
- **AC-6:** Given the 3D viewer is streaming, when the connection drops mid-session, then the system detects the disconnection, shows a reconnection spinner for up to 10 seconds, and falls back to 2D if reconnection fails.
- **AC-7:** Given the 3D viewer is active, when the user uses WASD keys or mouse, then the camera navigates the 3D environment with standard FPS-style controls (forward, backward, strafe, look).
- **AC-8:** Given the web app is running, when the user toggles between 2D and 3D views, then the switch happens without page reload and preserves the currently selected agent.
- **AC-9:** Given the UE5 backend WebSocket channel, when the web app sends a `focus_agent` command with an agent ID, then the UE5 camera smoothly flies to and frames the specified agent avatar.
- **AC-10:** Given a new agent is registered while the 3D viewer is active, when the agent creation event arrives via WebSocket, then a new 3D avatar spawns in the UE5 scene within 2 seconds.

**Technical Requirements:**

- Signaling server: Node.js-based signaling server (cirrus or equivalent) for WebRTC negotiation
- STUN/TURN: STUN server for NAT traversal; TURN server configuration for restrictive networks
- WebSocket protocol: Dedicated `/ws/ue5` endpoint for UE5 command channel, separate from the existing `/ws/live` event channel
- Docker: UE5 Pixel Streaming runs as a separate container (not in the main docker-compose; requires GPU host)
- Configuration: `UE5_SIGNALING_URL`, `UE5_STREAM_URL`, `UE5_ENABLED` environment variables
- Bandwidth: Target 1080p at 30fps; adaptive bitrate based on client bandwidth

### OAV-302: OpenTrace Integration (Visualize Distributed Traces)

**User Story:** As an AI engineer, I want to see distributed trace data from OpenTrace rendered as interactive waterfall diagrams inside OpenAgentVisualizer, so that I can correlate agent visual states with detailed trace timelines without switching tools.

**Description:** OpenTrace is the Open* Suite observability platform that stores and queries distributed traces at scale. OpenAgentVisualizer already ingests OTLP spans directly, but for production deployments, traces are routed through OpenTrace for persistent storage and analysis. This integration pulls trace data from OpenTrace's API and renders it as waterfall diagrams in a new "Traces" panel on the agent detail page and a dedicated "Trace Explorer" page.

**Acceptance Criteria:**

- **AC-1:** Given OpenTrace is configured and reachable, when a user opens the "Traces" tab on an agent detail page, then the system fetches the last 20 traces for that agent from OpenTrace and displays them as a list sorted by timestamp (newest first).
- **AC-2:** Given a list of traces is displayed, when the user clicks on a trace, then an interactive waterfall diagram renders showing all spans in the trace with timing bars, service names, operation names, duration, and status codes.
- **AC-3:** Given a waterfall diagram is rendered, when the user hovers over a span bar, then a tooltip shows span attributes: trace_id, span_id, parent_span_id, service, operation, duration_ms, status, and up to 10 custom attributes.
- **AC-4:** Given the user navigates to the "Trace Explorer" page, when the page loads, then a search interface allows filtering traces by: time range, agent ID, service name, minimum duration, and error status.
- **AC-5:** Given OpenTrace is unreachable, when the system attempts to fetch traces, then a graceful error message displays: "OpenTrace connection unavailable. Showing locally ingested spans only." The locally stored spans render as a simplified span list.
- **AC-6:** Given an OpenTrace connection is configured, when the backend starts, then it validates the connection on startup and logs the connection status. The health endpoint (`/health`) includes OpenTrace connectivity status.
- **AC-7:** Given traces are displayed, when the user clicks "View in OpenTrace", then a new tab opens with the trace deep-linked in the OpenTrace UI.

**Integration Specification:**

- **Connection:** HTTP REST API to OpenTrace backend
- **Authentication:** API key stored in `OPENTRACE_API_KEY` environment variable
- **Base URL:** `OPENTRACE_BASE_URL` environment variable (e.g., `http://opentrace:8000/api`)
- **Endpoints consumed:**
  - `GET /api/traces?agent_id={id}&limit={n}` -- list traces for agent
  - `GET /api/traces/{trace_id}` -- get full trace with spans
  - `GET /api/traces/search?service={name}&min_duration={ms}&error={bool}&start={iso}&end={iso}` -- search traces
- **Data model mapping:**
  - OpenTrace `Trace` -> OAV `TraceView` (trace_id, root_service, root_operation, duration_ms, span_count, error_count, started_at)
  - OpenTrace `Span` -> OAV `SpanView` (span_id, parent_span_id, service, operation, duration_ms, status, attributes)
- **Caching:** Trace list cached in Redis for 30 seconds (key: `opentrace:agent:{id}:traces`). Full trace cached for 5 minutes (key: `opentrace:trace:{trace_id}`).
- **Error handling:** Circuit breaker pattern -- after 3 consecutive failures, stop attempting for 60 seconds, then retry. Log all failures. Never crash the main application due to OpenTrace unavailability.
- **New backend files:**
  - `src/backend/app/services/opentrace_service.py` -- HTTP client, caching, circuit breaker
  - `src/backend/app/routers/integrations/opentrace.py` -- API endpoints
  - `src/backend/app/schemas/opentrace.py` -- Pydantic models
- **New frontend files:**
  - `src/frontend/src/pages/TraceExplorerPage.tsx` -- dedicated trace search page
  - `src/frontend/src/components/traces/TraceWaterfall.tsx` -- waterfall diagram component
  - `src/frontend/src/components/traces/TraceList.tsx` -- trace list component
  - `src/frontend/src/api/opentrace.ts` -- API client

### OAV-303: OpenMesh Integration (Workflow Topology Visualization)

**User Story:** As a platform engineer, I want to see multi-agent mesh network topologies from OpenMesh rendered as interactive workflow graphs inside OpenAgentVisualizer, so that I can understand how agents are connected and how messages flow through the mesh.

**Description:** OpenMesh is the Open* Suite multi-agent mesh networking product. It manages the communication topology between agents -- which agents can talk to each other, message routing, and workflow definitions. This integration pulls mesh topology data from OpenMesh and renders it as an enhanced ReactFlow graph in a new "Mesh Topology" page. The visualization extends the existing OAV-203 topology graph with mesh-specific data: message flow rates, routing paths, and agent connectivity status.

**Acceptance Criteria:**

- **AC-1:** Given OpenMesh is configured and reachable, when a user opens the "Mesh Topology" page, then the system fetches the current mesh topology and renders it as a ReactFlow directed graph with agents as nodes and communication channels as edges.
- **AC-2:** Given a mesh topology is rendered, when the user hovers over an edge, then a tooltip shows: message count (last hour), average latency, error rate, and the protocol (gRPC/HTTP/WebSocket).
- **AC-3:** Given a mesh topology is rendered, when the user hovers over a node, then a tooltip shows: agent name, agent status, connected peers count, messages sent/received (last hour), and mesh role (producer/consumer/router).
- **AC-4:** Given real-time mesh events arrive via WebSocket, when a new agent joins the mesh, then the node appears in the graph with an animated entry effect. When an agent disconnects, the node fades and shows a "disconnected" visual indicator.
- **AC-5:** Given mesh topology data includes message flow rates, when the graph renders, then edge thickness and animation speed reflect the relative message volume (thicker/faster = more messages).
- **AC-6:** Given OpenMesh is unreachable, when the system attempts to fetch topology data, then the page shows the locally computed agent relationship graph (from OAV-223, Sprint 2) with a banner: "OpenMesh unavailable -- showing local topology."
- **AC-7:** Given the mesh topology page is active, when the user clicks an agent node, then the agent detail sidebar opens (consistent with 2D canvas agent click behavior).

**Integration Specification:**

- **Connection:** HTTP REST API to OpenMesh backend
- **Authentication:** API key in `OPENMESH_API_KEY` environment variable
- **Base URL:** `OPENMESH_BASE_URL` environment variable
- **Endpoints consumed:**
  - `GET /api/mesh/topology?workspace_id={id}` -- full mesh topology (nodes + edges)
  - `GET /api/mesh/agents/{id}/connections` -- agent's direct connections
  - `GET /api/mesh/stats?workspace_id={id}&period={1h|24h|7d}` -- mesh-wide statistics
  - `WS /ws/mesh/events` -- real-time mesh topology change events
- **Data model mapping:**
  - OpenMesh `MeshNode` -> OAV `MeshNodeView` (agent_id, agent_name, role, status, connected_peers, messages_sent, messages_received)
  - OpenMesh `MeshEdge` -> OAV `MeshEdgeView` (source_agent_id, target_agent_id, protocol, message_count, avg_latency_ms, error_rate)
- **Caching:** Topology snapshot cached in Redis for 60 seconds (key: `openmesh:workspace:{id}:topology`). Stats cached for 5 minutes.
- **Error handling:** Same circuit breaker pattern as OpenTrace (3 failures -> 60s backoff).
- **Real-time sync:** Subscribe to OpenMesh WebSocket for topology change events. Map events to local graph updates.
- **New backend files:**
  - `src/backend/app/services/openmesh_service.py`
  - `src/backend/app/routers/integrations/openmesh.py`
  - `src/backend/app/schemas/openmesh.py`
- **New frontend files:**
  - `src/frontend/src/pages/MeshTopologyPage.tsx`
  - `src/frontend/src/components/mesh/MeshGraph.tsx` -- enhanced ReactFlow graph
  - `src/frontend/src/components/mesh/MeshStatsPanel.tsx`
  - `src/frontend/src/api/openmesh.ts`

### OAV-304: OpenMind Integration (Knowledge Graph Visualization)

**User Story:** As an AI researcher, I want to explore the knowledge graph maintained by OpenMind, rendered as an interactive force-directed graph inside OpenAgentVisualizer, so that I can understand the relationships and knowledge structures that agents rely on.

**Description:** OpenMind is the Open* Suite knowledge graph product. It stores entities, relationships, and embeddings that agents use for memory and reasoning. This integration pulls knowledge graph data from OpenMind and renders it as a force-directed graph visualization using ReactFlow with custom node types for different entity categories (concepts, facts, agent memories, embeddings).

**Acceptance Criteria:**

- **AC-1:** Given OpenMind is configured and reachable, when a user opens the "Knowledge Graph" page, then the system fetches the knowledge graph for the workspace and renders it as an interactive force-directed graph.
- **AC-2:** Given a knowledge graph is rendered, when the user sees entity nodes, then different entity types are visually distinct: concepts (blue circles), facts (green rectangles), agent memories (purple diamonds), embeddings (orange hexagons).
- **AC-3:** Given a knowledge graph is rendered, when the user clicks on an entity node, then a detail panel shows: entity name, type, description, creation date, related agents, and the 5 most related entities.
- **AC-4:** Given the knowledge graph page is active, when the user types in the search box, then the graph highlights nodes matching the query and dims non-matching nodes, with a debounced search (300ms delay).
- **AC-5:** Given a knowledge graph is rendered, when the user hovers over a relationship edge, then a tooltip shows: relationship type, strength/weight, and the agents that contributed to this relationship.
- **AC-6:** Given OpenMind is unreachable, when the system attempts to fetch graph data, then the page shows an empty state with: "OpenMind connection unavailable. Knowledge graph requires OpenMind to be running."
- **AC-7:** Given the knowledge graph has more than 200 entities, when the page loads, then the graph renders only the 200 highest-relevance entities with a "Load more" control to expand.

**Integration Specification:**

- **Connection:** HTTP REST API to OpenMind backend
- **Authentication:** API key in `OPENMIND_API_KEY` environment variable
- **Base URL:** `OPENMIND_BASE_URL` environment variable
- **Endpoints consumed:**
  - `GET /api/graph?workspace_id={id}&limit={n}&offset={n}` -- paginated knowledge graph
  - `GET /api/graph/entities/{id}` -- entity detail with relationships
  - `GET /api/graph/search?q={query}&workspace_id={id}` -- search entities
  - `GET /api/graph/agents/{id}/knowledge` -- knowledge associated with an agent
- **Data model mapping:**
  - OpenMind `Entity` -> OAV `KnowledgeNodeView` (entity_id, name, type, description, created_at, relevance_score)
  - OpenMind `Relationship` -> OAV `KnowledgeEdgeView` (source_entity_id, target_entity_id, relationship_type, weight, contributing_agents)
- **Caching:** Graph snapshot cached in Redis for 5 minutes (key: `openmind:workspace:{id}:graph`). Entity detail cached for 10 minutes.
- **Error handling:** Circuit breaker (3 failures -> 60s backoff). No local fallback -- OpenMind data has no local equivalent.
- **New backend files:**
  - `src/backend/app/services/openmind_service.py`
  - `src/backend/app/routers/integrations/openmind.py`
  - `src/backend/app/schemas/openmind.py`
- **New frontend files:**
  - `src/frontend/src/pages/KnowledgeGraphPage.tsx`
  - `src/frontend/src/components/knowledge/KnowledgeGraph.tsx`
  - `src/frontend/src/components/knowledge/EntityDetailPanel.tsx`
  - `src/frontend/src/api/openmind.ts`

**User Story:** As a security engineer, I want to see the security posture of my AI agents as reported by OpenShield, rendered as a compliance dashboard inside OpenAgentVisualizer, so that I can monitor data privacy, policy compliance, and threat indicators alongside agent operational data.

- **Error handling:** Circuit breaker (3 failures -> 60s backoff). No local fallback.

### OAV-306: CLI Plugin for Claude Code / Codex

**User Story:** As a developer using Claude Code or Codex in the terminal, I want a CLI plugin that shows agent status, streams events, triggers actions, and renders metrics as ASCII charts, so that I can monitor and control agents without leaving my terminal workflow.

**Description:** The CLI plugin is a standalone Python package (`oav-cli`) that communicates with the OpenAgentVisualizer backend via its REST API and WebSocket endpoint. It provides a rich terminal experience using the `rich` library for tables, panels, and live displays, and `asciichartpy` for ASCII metric charts. The plugin authenticates via API key (not JWT) for non-interactive use.

**Command Specification:**

```
oav status                         # Show all agents with status, level, XP
oav status <agent_id>              # Show detailed status for one agent
oav stream                         # Stream all real-time events to terminal
oav stream --agent <agent_id>      # Stream events for a specific agent
oav stream --type <event_type>     # Filter by event type
oav start <agent_id>               # Send start command to agent
oav stop <agent_id>                # Send stop command to agent
oav metrics                        # Show workspace metrics summary
oav metrics --chart                # Show ASCII chart of metrics over time
oav metrics --agent <agent_id>     # Show per-agent metrics
oav leaderboard                    # Show gamification leaderboard
oav topology                       # Show agent topology as ASCII tree
oav config set <key> <value>       # Set configuration (url, api_key)
oav config show                    # Show current configuration
oav health                         # Check backend connectivity
```

**Acceptance Criteria:**

- **AC-1:** Given the CLI is installed and configured with a valid API key and URL, when the user runs `oav status`, then a formatted table displays all agents in the workspace with columns: Name, Status, Level, XP, Last Event, showing data from the backend API.
- **AC-2:** Given the CLI is configured, when the user runs `oav stream`, then a live-updating terminal display shows real-time events as they arrive via WebSocket, with each event showing: timestamp, agent name, event type, and summary.
- **AC-3:** Given the CLI is configured, when the user runs `oav stream --agent <id>`, then only events for the specified agent appear in the stream, using WebSocket room subscription.
- **AC-4:** Given the CLI is configured, when the user runs `oav metrics --chart`, then an ASCII line chart renders showing token usage and cost over the last 24 hours, using the metrics API data.
- **AC-5:** Given the CLI is configured, when the user runs `oav leaderboard`, then a formatted table shows the top 10 agents ranked by XP with: Rank, Name, Level, XP, and Achievement Count.
- **AC-6:** Given the CLI is configured, when the user runs `oav topology`, then an ASCII tree diagram shows agent relationships based on the relationship graph API.
- **AC-7:** Given the backend is unreachable, when the user runs any command, then the CLI shows a clear error message: "Cannot connect to OpenAgentVisualizer at {url}. Check your configuration with `oav config show`."
- **AC-8:** Given the CLI is not yet configured, when the user runs `oav config set url <url>` and `oav config set api_key <key>`, then the configuration is stored in `~/.oav/config.toml` and subsequent commands use these values.
- **AC-9:** Given the CLI is installed, when the user runs `oav health`, then the CLI checks backend connectivity and reports: backend status (up/down), WebSocket status (up/down), database status (from health endpoint), and API version.

**Technical Requirements:**

- **Package:** Python package `oav-cli`, installable via `pip install oav-cli`
- **Dependencies:** `httpx` (async HTTP), `websockets` (WebSocket client), `rich` (terminal formatting), `asciichartpy` (charts), `click` (CLI framework), `tomli`/`tomli_w` (config file)
- **Config file:** `~/.oav/config.toml` with keys: `url`, `api_key`, `default_workspace_id`
- **Authentication:** Uses `X-API-Key` header (existing API key endpoints from Sprint 1)
- **Source location:** `src/cli/` directory in the monorepo
- **New files:**
  - `src/cli/pyproject.toml` -- package metadata
  - `src/cli/oav_cli/__init__.py`
  - `src/cli/oav_cli/main.py` -- Click CLI entry point

  - `src/cli/oav_cli/commands/metrics.py`
  - `src/cli/oav_cli/commands/leaderboard.py`
  - `src/cli/oav_cli/commands/topology.py`
  - `src/cli/oav_cli/commands/config.py`

### 6.2 Circuit Breaker Specification

All integration services use a shared circuit breaker implementation:

```python
class CircuitBreaker:
    """States: CLOSED (normal), OPEN (failing), HALF_OPEN (testing recovery)"""
    failure_threshold: int = 3        # failures before opening
    recovery_timeout: int = 60        # seconds before half-open
    success_threshold: int = 1        # successes in half-open to close

    async def call(self, func, *args, **kwargs):
        # CLOSED: call func, track failures
        # OPEN: raise CircuitOpenError immediately
        # HALF_OPEN: call func, close on success, open on failure
```

## 7. UE5 Pixel Streaming Requirements

### 7.1 Architecture

```
Browser (React SPA)
  |
  |-- WebRTC video stream (Pixel Streaming)
  |-- DataChannel (UE5 commands: select agent, move camera)
  |
Signaling Server (Node.js)
  |
  |-- WebSocket signaling (SDP exchange, ICE candidates)
  |
UE5 Application (GPU Server)
  |
  |-- Pixel Streaming Plugin (built-in UE5)
  |-- Custom GameMode: AgentWorldGameMode
  |-- WebSocket client -> FastAPI /ws/ue5 for agent state sync
  |
FastAPI Backend
  |
  |-- /ws/ue5 endpoint: bidirectional UE5 state sync
  |-- existing /ws/live: frontend event stream
```

### 7.2 WebSocket Protocol (`/ws/ue5`)

**Web App -> UE5 (via backend relay):**

| Command | Payload | Description |
|---------|---------|-------------|
| `focus_agent` | `{ agent_id: string }` | Camera flies to agent |
| `select_agent` | `{ agent_id: string }` | Highlight agent in 3D |
| `deselect_all` | `{}` | Remove all highlights |
| `set_view_mode` | `{ mode: "overview" \| "agent_follow" \| "free_camera" }` | Camera mode |
| `sync_agents` | `{ agents: AgentState[] }` | Full agent state sync |

**UE5 -> Web App (via backend relay):**

| Event | Payload | Description |
|-------|---------|-------------|
| `agent_clicked` | `{ agent_id: string }` | User clicked agent in 3D |
| `camera_moved` | `{ position: Vec3, rotation: Vec3 }` | Camera state update |
| `scene_ready` | `{ agent_count: int }` | UE5 scene loaded |
| `agent_spawned` | `{ agent_id: string }` | 3D avatar created |

### 7.3 Fallback Strategy

1. On page load, attempt WebRTC connection to signaling server (timeout: 3 seconds)
2. If connection fails or `UE5_ENABLED=false`, render 2D PixiJS canvas immediately
3. Show banner: "3D viewer unavailable -- showing 2D view. [Configure 3D]"
4. "Configure 3D" links to Settings page Integrations tab
5. If connection drops during session, show reconnection spinner (10 seconds), then fall back to 2D

### 7.4 Resource Requirements

| Component | Minimum Spec | Recommended |
|-----------|-------------|-------------|
| UE5 Server GPU | NVIDIA T4 (16GB VRAM) | NVIDIA A10G (24GB VRAM) |
| UE5 Server CPU | 8 cores | 16 cores |
| UE5 Server RAM | 16 GB | 32 GB |
| Network bandwidth | 10 Mbps per concurrent viewer | 20 Mbps per viewer |
| Signaling server | 1 vCPU, 512MB RAM | 2 vCPU, 1GB RAM |

## 11. Gamification Enhancements (Sprint 3)

### 11.1 XP Decay (New)

Sprint 2 noted XP decay as a Sprint 3 consideration. Sprint 3 introduces optional XP decay to keep leaderboards dynamic:

- **Decay rate:** 1% of total XP per day of agent inactivity (no events for 24 hours)
- **Minimum floor:** XP cannot decay below the threshold for the agent's current level minus one (agents cannot lose more than one level from decay)
- **Configuration:** Workspace-level setting, disabled by default. Toggle in Settings page.
- **Implementation:** Celery beat task runs daily at 00:00 UTC, applies decay to inactive agents, emits `xp_decay` WebSocket events.
