## 1. Executive Summary

Sprint 1 delivered a complete backend platform: authentication, agent CRUD with gamification fields, event ingestion via TimescaleDB hypertables, OTLP trace receiver, WebSocket live streaming, Redis-backed Celery task queue, gamification XP engine, metrics aggregation, alert management, and a 13-file test suite. The backend is production-capable but invisible -- there is no frontend rendering, no visualization, and no user-facing dashboard.

Sprint 2 transforms OpenAgentVisualizer from a headless API into a gamified virtual world. The sprint has four objectives:

1. **Canvas Rendering Engine** -- Build a PixiJS-powered 2D virtual world where AI agents appear as animated avatars, transition through XState-driven state machines, connect via ReactFlow topology graphs, and animate with GSAP/Rive. The canvas must sustain 60fps with 50 concurrent agent sprites.

2. **Dashboard Pages** -- Deliver seven pages (main dashboard, agent detail, leaderboard, analytics, alerts, session viewer, settings) that surface the backend data through charts, timelines, and interactive controls.

3. **Backend Enhancements** -- Extend WebSocket with per-agent room filtering, add event replay for historical playback, compute agent relationship graphs, implement an achievement system, and add continuous metric aggregates.

4. **Quality and DevOps** -- Canvas rendering tests, XState transition tests, Vitest component coverage, WebSocket stress tests, Playwright E2E, GitHub Actions CI, production Dockerfiles, and Prometheus metrics.

The gamification angle is central: agents earn XP, level up with visual effects, unlock achievements displayed on their avatar, compete on leaderboards, and exist in a spatial world with animated state transitions. This is not a traditional monitoring dashboard -- it is a game-like environment where observing AI agents is engaging and rewarding.

## 3. Gamification Mechanics Design

### 3.1 XP System (extends Sprint 1)

Sprint 1 established XP transactions (`POST /api/gamification/xp`). Sprint 2 defines the trigger rules and balancing.

**XP Award Triggers:**

| Trigger Event | XP Amount | Cooldown | Description |
|---------------|-----------|----------|-------------|
| `task_completed` | 100 XP | None | Agent completes a task successfully |
| `task_completed_fast` | 150 XP | None | Task completed in < 50th percentile latency for that agent |
| `session_completed` | 50 XP | None | Agent session ends normally |
| `error_recovered` | 200 XP | 5 min | Agent transitions from error state back to active |
| `streak_3` | 300 XP (bonus) | Per streak | 3 consecutive tasks without error |
| `streak_5` | 500 XP (bonus) | Per streak | 5 consecutive tasks without error |
| `first_event_of_day` | 25 XP | 24 hours | First event ingested for agent that day |
| `cost_efficient` | 75 XP | Per task | Task cost is < 50th percentile cost for that task type |
| `loop_avoided` | 150 XP | None | Agent completes workflow that previously triggered loop detection |

**XP Decay:** No XP decay in Sprint 2. Consider for Sprint 3 to keep leaderboards dynamic.

**Anti-Cheat:** Gamification service must be idempotent -- duplicate events with the same event ID must not award duplicate XP. Rate limit: max 50 XP transactions per agent per minute.

### 3.2 Leveling System

| Level | XP Required (cumulative) | Title | Visual Effect |
|-------|--------------------------|-------|---------------|
| 1 | 0 | Novice | Gray avatar ring |
| 2 | 500 | Apprentice | Blue avatar ring |
| 3 | 1,500 | Operative | Green avatar ring |
| 4 | 3,500 | Specialist | Purple avatar ring |
| 5 | 7,000 | Expert | Gold avatar ring |
| 6 | 12,000 | Master | Gold ring + particle trail |
| 7 | 20,000 | Grandmaster | Gold ring + particle trail + crown icon |
| 8 | 35,000 | Legend | Animated aura + crown + unique sprite tint |
| 9 | 60,000 | Mythic | Full animated avatar transformation |
| 10 | 100,000 | Transcendent | Custom particle system + screen-wide announcement on level-up |

**Level-up formula:** `required_xp(level) = round(500 * (level - 1) ^ 1.8)` (adjusted to match table above).

**Level-up event:** When an agent levels up, the backend emits a `level_up` event via WebSocket. The frontend plays a GSAP animation on the agent avatar (scale pulse + particle burst) and displays a toast notification.

### 3.3 Achievement System (OAV-224)

Achievements are persistent badges awarded when conditions are met. Each achievement has an icon, a name, a description, and an XP bonus.

**Achievement Definitions:**

| ID | Name | Condition | XP Bonus | Icon |
|----|------|-----------|----------|------|
| ACH-001 | First Steps | Agent completes first task | 50 | Footprint |
| ACH-002 | Centurion | Agent completes 100 tasks | 500 | Shield |
| ACH-003 | Speed Demon | Agent completes 10 tasks with < 1s latency | 300 | Lightning |
| ACH-004 | Penny Pincher | Agent completes 50 tasks under median cost | 400 | Coin |
| ACH-005 | Iron Will | Agent recovers from error 10 times | 350 | Anvil |
| ACH-006 | Marathon Runner | Agent has 24+ hours of total session time | 500 | Clock |
| ACH-007 | Team Player | Agent appears in 5+ relationship edges (graph) | 250 | Handshake |
| ACH-008 | Perfect Streak | Agent completes 10 consecutive tasks without error | 600 | Star |
| ACH-009 | Night Owl | Agent completes 50 tasks between 22:00-06:00 UTC | 200 | Moon |
| ACH-010 | Trailblazer | First agent in workspace to reach Level 5 | 1000 | Flag |

**Achievement storage:** New `achievements` table:
- `id` (VARCHAR PK)
- `workspace_id` (FK)
- `agent_id` (FK)
- `achievement_id` (VARCHAR, e.g., "ACH-001")
- `achievement_name` (VARCHAR)
- `xp_bonus` (INTEGER)
- `unlocked_at` (TIMESTAMPTZ)

**Achievement display:** Badges appear below agent avatars in the PixiJS canvas and on the agent detail page.

### 3.4 Leaderboard Rules

The leaderboard (`GET /api/gamification/leaderboard`) ranks agents within a workspace by XP. Sprint 2 adds:

- **Time-scoped leaderboards:** `?period=daily|weekly|monthly|all_time`
- **Category leaderboards:** `?category=xp|tasks|cost_efficiency|streaks`
- **Leaderboard page** shows top 50 agents with rank, avatar, level, XP, achievement count, and trend arrow (up/down/same vs. previous period)
- **Rank change events:** When an agent moves up or down in rank, a WebSocket event is emitted so the canvas can animate rank change effects

### OAV-201: PixiJS 2D Agent Visualization Canvas

**User Story:** As an AI team lead, I want to see my AI agents rendered as animated avatars in a 2D virtual world canvas, so that I can spatially monitor their activity and status at a glance.

**Description:** The PixiJS canvas is the heart of OpenAgentVisualizer. It renders each registered agent as a sprite-based avatar in a pannable, zoomable 2D world. Agents are positioned using a force-directed layout. Each avatar displays the agent's name, level ring (colored by level), status indicator (colored dot), and achievement badges. The canvas uses PixiJS 8.1 via @pixi/react 7.1, rendered inside a React component.

**Acceptance Criteria:**

- **AC-1:** Given 50 registered agents in a workspace, when the canvas loads, then all 50 agents render as distinct avatar sprites at 60fps or higher (measured via PixiJS Ticker).
- **AC-2:** Given an agent with level 5 and status "active", when the canvas renders that agent, then the avatar displays a gold ring (level 5 indicator) and a green status dot.
- **AC-3:** Given the canvas is rendered, when the user scrolls the mouse wheel, then the canvas zooms in/out smoothly with min zoom 0.25x and max zoom 4x.
- **AC-4:** Given the canvas is rendered, when the user clicks and drags on empty space, then the canvas pans in the drag direction.
- **AC-5:** Given the canvas is rendered, when the user clicks on an agent avatar, then a tooltip appears showing agent name, level, XP, status, and current task (if any).
- **AC-6:** Given the canvas is rendered, when the user double-clicks an agent avatar, then the app navigates to the agent detail page (OAV-212).
- **AC-7:** Given agents have achievements, when the canvas renders, then achievement badge icons appear below the avatar (max 3 visible, "+N" for overflow).
- **AC-8:** Given an agent levels up (level_up WebSocket event received), when the event is processed, then the avatar plays a scale-pulse animation (1.0x to 1.3x to 1.0x over 600ms) and emits a particle burst.
- **AC-9:** Given the canvas is mounted, when the React component unmounts, then all PixiJS resources (textures, sprites, containers) are properly destroyed with no memory leaks.
- **AC-10:** Given agents exist, when the canvas initializes, then agents are positioned using a force-directed layout algorithm that avoids overlap and distributes agents evenly.

### OAV-202: Agent State Machine Visualization with XState

**User Story:** As an AI engineer, I want to see the current state of each agent rendered as a visual state machine, so that I can understand what phase of execution an agent is in and trace its state transitions.

**Description:** Each agent is governed by an XState 5.13 finite state machine with states: `idle`, `active`, `waiting`, `error`, `complete`. The FSM drives both the avatar visuals (color, animation) and a state diagram overlay. Users can inspect the state machine for any agent and see transition history.

**Acceptance Criteria:**

- **AC-1:** Given an agent in state "idle", when the canvas renders, then the avatar has a gray tint and a slow breathing animation (GSAP scale 1.0 to 1.02, 2s loop).
- **AC-2:** Given an agent in state "active", when the canvas renders, then the avatar has a green tint, a bright glow effect, and a faster pulse animation.
- **AC-3:** Given an agent in state "error", when the canvas renders, then the avatar has a red tint and a shake animation (GSAP x-offset +/- 3px, 300ms).
- **AC-4:** Given an agent in state "waiting", when the canvas renders, then the avatar has an amber tint and a clock icon overlay.
- **AC-5:** Given an agent in state "complete", when the canvas renders, then the avatar has a blue tint and a checkmark icon overlay.
- **AC-6:** Given an agent transitions from "idle" to "active", when the transition event arrives via WebSocket, then the XState machine transitions, the avatar animates from gray to green tint over 400ms, and the glow effect fades in.
- **AC-7:** Given the user clicks "View State Machine" on an agent, when the overlay opens, then a visual state diagram renders showing all 5 states, transition arrows, the current state highlighted, and the last 10 transitions listed with timestamps.
- **AC-8:** Given the XState machine is running, when the component unmounts, then the machine is stopped and all subscriptions are cleaned up (no memory leaks verified by test).
- **AC-9:** Given 50 agents each with their own XState machine instance, when all are running simultaneously, then memory usage does not grow unboundedly over a 60-second observation period.

### OAV-203: Interactive Topology Graph with ReactFlow

**User Story:** As an AI team lead, I want to see a topology graph showing how my agents relate to each other (data flow, delegation, shared sessions), so that I can understand the structure of my multi-agent system.

**Description:** ReactFlow 11.11 renders an interactive node-edge graph where each node is an agent and edges represent relationships (computed by OAV-223). Users can drag nodes, zoom, and click edges to see relationship details.

**Acceptance Criteria:**

- **AC-1:** Given agents with computed relationships (from OAV-223), when the topology page loads, then each agent appears as a ReactFlow node with its name, level badge, and status color.
- **AC-2:** Given two agents have a "delegates_to" relationship, when the graph renders, then a directed edge with an arrow connects the source to the target, labeled "delegates".
- **AC-3:** Given two agents share a session, when the graph renders, then an undirected dashed edge connects them, labeled "shared session".
- **AC-4:** Given the graph is rendered, when the user drags a node, then the node moves and connected edges update in real-time.
- **AC-5:** Given the graph is rendered, when the user clicks an edge, then a popover shows: relationship type, event count, first/last interaction timestamps.
- **AC-6:** Given the graph is rendered, when the user clicks a node, then the node expands to show agent stats (level, XP, task count, status).
- **AC-7:** Given new relationship data arrives via WebSocket, when processed, then the graph adds/removes edges without full re-render (incremental update).
- **AC-8:** Given 30+ agents with 50+ edges, when the graph renders, then the auto-layout completes in < 2 seconds and nodes do not overlap.

### OAV-204: Animated Agent Transitions with GSAP

**User Story:** As a user of the virtual world, I want agent state changes to be visually animated (smooth color transitions, scale effects, particle bursts), so that the visualization feels alive and state changes are noticeable.

**Description:** GSAP 3.12 powers all avatar animations in the PixiJS canvas. Animations are triggered by XState transitions and gamification events.

**Acceptance Criteria:**

- **AC-1:** Given an agent transitions from any state to "active", when the transition occurs, then GSAP animates the avatar tint from current color to green over 400ms with an ease-out curve.
- **AC-2:** Given an agent transitions to "error", when the transition occurs, then GSAP plays a shake animation (x-offset oscillation, 3 cycles, 300ms total) and tint shifts to red.
- **AC-3:** Given an agent levels up, when the level_up event arrives, then GSAP plays: (a) scale pulse 1.0 to 1.3 to 1.0 over 600ms, (b) particle burst of 20 particles radiating outward over 800ms, (c) floating "+N XP" text that rises and fades over 1000ms.
- **AC-4:** Given an agent earns an achievement, when the achievement event arrives, then GSAP plays: badge icon slides in from bottom of avatar, badge glows for 500ms, then settles into badge row.
- **AC-5:** Given multiple animations trigger simultaneously on the same avatar, when processed, then GSAP timeline ensures animations queue rather than conflict (no visual glitches).
- **AC-6:** Given 50 agents with concurrent animations, when all play simultaneously, then frame rate stays above 55fps.

### OAV-205: Rive Animations for Agent States

**User Story:** As a user, I want rich vector animations (Rive) for special agent events (level-up celebrations, achievement unlocks, critical errors), so that the virtual world has high-quality, engaging visual feedback.

**Description:** @rive-app/canvas 2.21 renders Rive (.riv) animation files for premium visual effects. Rive complements GSAP -- GSAP handles simple property tweens while Rive handles complex multi-element animations.

**Acceptance Criteria:**

- **AC-1:** Given a Rive animation file exists for "level_up_celebration", when an agent levels up, then the Rive animation plays centered on the agent avatar for 2 seconds.
- **AC-2:** Given a Rive animation file exists for "achievement_unlock", when an agent earns an achievement, then the Rive animation plays at the achievement badge position for 1.5 seconds.
- **AC-3:** Given a Rive animation file exists for "critical_error", when an agent enters error state with severity "critical", then the Rive animation plays as a full-avatar overlay for 1 second.
- **AC-4:** Given no .riv file is available for an animation trigger, when the trigger fires, then the system gracefully falls back to the GSAP animation (OAV-204) with no errors.
- **AC-5:** Given Rive animations are loaded, when the canvas component unmounts, then all Rive instances are properly disposed (no memory leaks).
- **AC-6:** Given the Rive animation is playing, when another animation trigger fires on the same agent, then the current Rive animation completes before the next one starts (queue, do not overlap).

### OAV-206: Real-Time Canvas Updates via WebSocket

**User Story:** As a user watching the virtual world, I want agent avatars to update in real-time as events arrive (state changes, XP awards, new tasks), so that the canvas reflects the live state of my agent system.

**Description:** The frontend WebSocket client subscribes to rooms (OAV-221) and dispatches incoming events to the PixiJS canvas, XState machines, Zustand stores, and dashboard components.

**Acceptance Criteria:**

- **AC-1:** Given the WebSocket connection is established, when a `state_change` event arrives for agent X, then the XState machine for agent X transitions within 100ms, and the canvas avatar updates within 500ms.
- **AC-2:** Given the WebSocket connection is established, when an `xp_awarded` event arrives, then the agent's XP in the Zustand store updates, the canvas shows a floating "+N XP" animation, and the leaderboard re-sorts if rank changed.
- **AC-3:** Given the WebSocket connection is established, when a `level_up` event arrives, then the level-up animation plays (OAV-204/OAV-205) and the avatar ring color updates.
- **AC-4:** Given the WebSocket connection is established, when an `achievement_unlocked` event arrives, then the achievement animation plays and the badge appears on the avatar.
- **AC-5:** Given the WebSocket connection drops, when the client detects disconnection, then it attempts reconnection with exponential backoff (1s, 2s, 4s, 8s, max 30s) and shows a "Reconnecting..." indicator in the UI.
- **AC-6:** Given the WebSocket reconnects after a disconnection, when the connection is re-established, then the client requests a full state sync from the server to reconcile any missed events.
- **AC-7:** Given 20 events arrive in a 100ms burst, when processed, then the canvas batches updates into a single render frame (no frame-per-event rendering).
- **AC-8:** Given the user subscribes to a specific agent room, when events arrive for other agents, then those events do not trigger canvas updates for the subscribed agent (filtering works).

### OAV-221: WebSocket Room Support (Per-Agent Filtering)

**User Story:** As a frontend client, I want to subscribe to WebSocket "rooms" for specific agents, so that I only receive events relevant to the agents I am viewing and reduce unnecessary network traffic.

**Description:** Extend the existing `/ws/live` WebSocket endpoint to support room-based subscriptions. Clients send a subscribe/unsubscribe message to join rooms. Events are routed to the correct rooms via Redis Pub/Sub.

**Acceptance Criteria:**

- **AC-1:** Given a WebSocket connection is established, when the client sends `{"action": "subscribe", "room": "agent:{agent_id}"}`, then the server adds the connection to that agent's room.
- **AC-2:** Given a client is subscribed to agent X's room, when an event for agent X is ingested, then only clients in agent X's room receive the event.
- **AC-3:** Given a client is subscribed to agent X's room, when an event for agent Y is ingested, then the client does NOT receive the event.
- **AC-4:** Given a client is subscribed to multiple rooms, when the client sends `{"action": "unsubscribe", "room": "agent:{agent_id}"}`, then the client is removed from that room but remains in other rooms.
- **AC-5:** Given a client subscribes to `{"action": "subscribe", "room": "workspace:{workspace_id}"}`, then the client receives ALL events for the entire workspace (broadcast room).
- **AC-6:** Given the WebSocket connection drops, when the server detects disconnection, then the client is removed from all rooms with no dangling references.
- **AC-7:** Given 100 concurrent WebSocket connections across 20 rooms, when events are published, then routing completes within 50ms per event.

### OAV-223: Agent Relationship Graph Computation

## 7. Non-Functional Requirements

### 7.1 Performance

- PixiJS canvas: 60fps with 50 agent sprites, each with tint, ring, status dot, and up to 3 badge icons
- Page load: < 3 seconds for dashboard, < 2 seconds for subsequent navigation (SPA)
- API response: < 500ms for all list endpoints, < 200ms for single resource endpoints
- WebSocket event delivery: < 2 seconds from backend ingestion to canvas render
- ReactFlow graph layout: < 2 seconds for 30 nodes and 50 edges

### 7.2 Accessibility

- All dashboard pages must meet WCAG 2.1 AA for non-canvas content
- Canvas content must have keyboard navigation (Tab between agents, Enter to select)
- Color choices must pass contrast ratio checks (4.5:1 for text, 3:1 for graphical elements)
- Screen reader announcements for state changes and alerts

### 7.3 Browser Support

- Chrome 120+, Firefox 120+, Safari 17+, Edge 120+
- WebGL2 required for PixiJS (graceful fallback message if not available)

### 7.4 Security

- All API calls use JWT Bearer authentication
- WebSocket connections authenticate via token in connection handshake
- API keys are hashed (never stored in plaintext)
- No sensitive data in WebSocket event payloads (filtered server-side)
